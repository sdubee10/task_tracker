# MyTaskFlow - 업무 관리 시스템

---

## 1. 프로젝트 개요

### 프로젝트 이름
**Task Flow** (타스크플로우)

### 결과물 형태
**웹앱** - SPA(Single Page Application) 기반 업무 관리 플랫폼

### 한 줄 요약
> **여러 채널에 흩어진 업무를 단일 플랫폼에 기록함으로써 나의 모든 업무 및 성과를 단일 시스템에 기록하여 내가 어떤 업무를 많이 수행하는지/잘하는지·어디서 성장할 수 있는지를 데이터로 증명하는 업무 관리 시스템**

---

## 2. 프로젝트 목적

### 🎯 핵심 목적: 단일 채널에 나의 모든 업무 수행량과 활동 내역을 기록하여, 노력과 성과 가치를 공정하게 증명하고, 다음 단계의 전문 역량 강화를 위한 맞춤형 인사이트 얻기

> **"내가 얼마나 일했는지, 무엇을 잘하는지, 어디서 성장할 수 있는지를 데이터로 증명"**

#### 문제 인식

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        현재 업무 환경의 문제점                            │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   📧 이메일        💬 메신저        📞 전화        📝 구두 요청           │
│      │               │               │               │                  │
│      └───────────────┴───────────────┴───────────────┘                  │
│                              │                                          │
│                              ▼                                          │
│                    ┌─────────────────┐                                  │
│                    │   업무 수행자   │                                  │
│                    └─────────────────┘                                  │
│                              │                                          │
│                              ▼                                          │
│              ┌───────────────────────────────┐                          │
│              │  ❌ 기록 없음 / 산발적 기록    │                          │
│              │  ❌ 업무량 증명 어려움         │                          │
│              │  ❌ 역량/성장 파악 불가        │                          │
│              │  ❌ 다른 사람에게 보여주기 어려움│                          │
│              └───────────────────────────────┘                          │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**여러 채널을 통해 업무를 수행하다 보면:**
- 📊 **업무량 파악 어려움**: 개인이 얼마나 많은 업무를 수행했는지/하고 있는지 정량화하기 어려움
- 🔍 **역량 분석 불가**: 어떤 업무를 잘하고, 어떤 부분이 부족한지 객관적으로 파악 어려움
- 📈 **성장 방향 불명확**: 다음 단계로 성장하기 위해 무엇을 해야 하는지 알기 어려움
- 👥 **성과 공유 어려움**: 개인의 성과를 다른 사람에게 객관적으로 알려주기 어려움

| 목적 | 설명 | TaskFlow 기능 |
|------|------|--------------|
| **📊 업무량 기록** | 단순/복잡한 모든 업무를 단일 채널에서 기록 | 신청서 작성 및 처리 이력 저장 |
| **🔍 역량 파악** | 어떤 업무를 잘하고 부족한지 객관적 분석 | 카테고리별 처리량, 평가 점수 분석 |
| **📈 성장 방향 제시** | 한 단계 더 성장하기 위한 방향 제시 | 강점/개선점 분석, 역량 평가 |
| **👥 성과 공유** | "이만큼 일했어요"를 데이터로 증명 | 업무 분석 대시보드, 처리 순위 |

### ✅ 업무 효율화
기존 업무 요청 프로세스의 불편함과 비효율을 개선합니다.

| 기존 문제점 | TaskFlow 해결책 |
|------------|----------------|
| 부서마다 다른 요청 양식 | 25종 컴포넌트로 표준화된 템플릿 시스템 |
| 이메일/메신저로 산발적 요청 | 통합 플랫폼에서 모든 요청 관리 |
| 업무 현황 파악 어려움 | 그래프 시각화로 실시간 현황 파악 |
| 담당자별 업무량 불균형 | 데이터 기반 업무량 분석 및 배정 |
| 업무 품질 관리 부재 | 요청자 평가 시스템으로 품질 관리 |

### ✅ 개인 성장 지원

#### 🎯 내가 수행한 업무 파악
- **처리 건수**: 총 몇 건의 업무를 처리했는지
- **카테고리별 분포**: 어떤 유형의 업무를 주로 처리했는지
- **처리 시간**: 평균 얼마나 빠르게 처리했는지

#### 📊 강점과 약점 분석
- **높은 평가 업무**: 어떤 업무에서 높은 평가를 받았는지
- **개선 필요 영역**: 어떤 부분에서 피드백이 있었는지
- **업무 특성 파악**: 나의 업무 스타일과 특징

#### 📈 성장 방향 제시
- **역량 발전 추이**: 시간에 따른 역량 변화
- **다음 단계 제안**: 더 도전할 수 있는 업무 영역
- **학습 포인트**: 개선이 필요한 구체적인 스킬

#### 👥 성과 공유
- **업무 분석 대시보드**: 한눈에 보는 나의 업무 성과
- **처리 순위**: 팀 내 기여도 확인
- **평가 요약**: 요청자들의 피드백 종합

### ✅ 문제 해결
신청서마다 다른 컴포넌트 구성으로 인한 **관계형 DB의 한계**를 **그래프 데이터베이스**로 해결하여, 유연한 데이터 저장과 관계 분석을 가능하게 합니다.

### ✅ AI 창작 실험
- **AI 기반 신청서 검증**: 제출 전 AI가 신청서 내용을 분석하여 누락/오류 피드백
- **Vibe Coding**: 전체 프로젝트를 AI(Claude)와의 대화를 통해 설계 및 구현

## 3. 주요 기능

### 🎯 핵심 기능 6가지

| # | 기능 | 설명 |
|---|------|------|
| 1 | **드래그 앤 드롭 폼 빌더** | 25종 컴포넌트를 드래그하여 신청서 양식 자유 설계 |
| 2 | **3단계 신청서 작성** | 템플릿 선택 → 내용 작성 → 검토 및 제출 프로세스 |
| 3 | **그래프 기반 관계 시각화** | 신청서-담당자 관계를 6가지 레이아웃으로 시각화 |
| 4 | **업무 분석 대시보드** | 개인/팀별 업무량, 처리 시간, 역량 평가 분석 |
| 5 | **AI 신청서 검증** | 제출 전 AI가 내용 검토 및 개선점 피드백 |
| 6 | **요청자 평가 시스템** | 업무 완료 후 기술/정성/정량적 평가 |

## 4. 사용 기술

### 🛠 기술 스택

| 분류 | 기술 | 용도 |
|------|------|------|
| **Frontend** | HTML5, CSS3, JavaScript (ES6+) | UI/UX 구현 |
| **차트** | Chart.js | 통계 시각화 |
| **그래프** | SVG | 관계 그래프 렌더링 |
| **아키텍처** | SPA (Single Page Application) | 페이지 전환 없는 UX |
| **라우팅** | Hash-based Router | 클라이언트 라우팅 |

| **데이터베이스** | Graph Database (Client-side) | 노드/엣지 기반 관계 관리 |
| **AI** | Claude (Anthropic) | Vibe Coding, 신청서 검증 로직 설계 |

| **아이콘** | SVG 인라인 아이콘 |

| **반응형** | 모바일, 태블릿, 데스크톱 대응 |

### 📱 화면별 사용자 흐름

#### 1️⃣ 로그인 → 대시보드

```
로그인 페이지 접속
    │
    ▼
이메일/비밀번호 입력 (또는 데모 계정 클릭)
    │
    ▼
인증 성공
    │
    ▼
대시보드 표시
├─ 통계 카드 (총 신청서, 완료, 처리 중, 대기)
├─ 상태별 차트
├─ 카테고리별 차트
└─ 최근 신청서 목록
```

#### 2️⃣ 신청서 작성

```
"신청서 작성" 클릭
    │
    ▼
Step 1: 템플릿 선택
├─ 카테고리 탭 클릭 (DBA, Frontend, Backend...)
├─ 템플릿 카드 클릭
└─ "다음" 버튼
    │
    ▼
Step 2: 내용 작성
├─ 동적 폼에 정보 입력
├─ (선택) "AI 검증" 버튼 → AI 피드백 확인
└─ "다음" 버튼
    │
    ▼
Step 3: 검토 및 제출
├─ 입력 내용 최종 확인
├─ (필요시) "이전" 버튼으로 수정
└─ "제출" 버튼
    │
    ▼
성공 모달 표시
├─ 신청서 ID 확인
├─ "대시보드로 이동" 또는
└─ "새 신청서 작성"
```

## 6. AI 활용 내역


### 🤖 Vibe Coding으로 전체 프로젝트 구현


┌─────────────────────────────────────────────────────────────────────────┐
│                         Bottom-Up 방식 (이 프로젝트)                     │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   1. 핵심 컴포넌트부터 구현 (폼 빌더, 드래그앤드롭)                        │
│                              ▼                                          │
│   2. 사용하면서 필요한 기능 추가 (템플릿, 테마)                            │
│                              ▼                                          │
│   3. 데이터 구조 발견 → DB 설계 (그래프 DB로 전환)                         │
│                              ▼                                          │
│   4. 전체 시스템으로 통합 (SPA, 분석 대시보드)                             │
│                                                                         │
│   ✅ 장점: 실제 필요에 따라 유연하게 진화                                  │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

#### 📋 실제 개발 진행 순서 (대화 히스토리 기반)

| 순서 | 개발 내용 | Bottom-Up 특징 |
|------|----------|---------------|
| 1 | **폼 빌더 컴포넌트** 개발 | 가장 작은 단위부터 시작 |
| 2 | **템플릿 10개 → 30개** 확장 | 필요에 따라 확장 |
| 3 | **테마 시스템** 추가 | 사용하면서 UX 개선 필요성 발견 |
| 4 | **그래프 시각화** 추가 | 관계 파악 필요성 발견 |
| 5 | **DB 설계 시도** → **관계형 DB 한계 발견** | 실제 데이터로 문제 발견 |
| 6 | **그래프 DB로 전환** | 문제 해결을 위한 구조 변경 |
| 7 | **SPA 통합** | 개별 페이지들을 하나로 통합 |
| 8 | **분석/평가 기능** 추가 | 축적된 데이터 활용 필요성 발견 |

#### 🎯 Bottom-Up을 선택한 핵심 이유

**1. 요구사항이 명확하지 않았다**
- 처음부터 "업무 기록 + 역량 분석 + 성과 증명"이라는 최종 목표가 있었던 게 아님
- **폼 빌더**를 만들다가 → **템플릿이 필요**하다는 걸 발견
- **템플릿**을 만들다가 → **각 신청서마다 컴포넌트가 다르다**는 걸 발견
- **DB 설계**하다가 → **관계형 DB로는 안 된다**는 걸 발견

**2. 신청서마다 컴포넌트가 다른 문제**

```
신청서 A: [제목, 텍스트, 날짜, 파일첨부]
신청서 B: [제목, 체크박스, 라디오, 서명]
신청서 C: [제목, 테이블, 평가척도, 결재라인]

→ 관계형 DB로 일괄 스키마 정의 불가능
→ 실제로 데이터를 다뤄보니 그래프 DB가 필요하다는 걸 발견
```

**3. Vibe Coding의 특성**
- AI와 대화하면서 **점진적으로** 기능을 추가
- 한 기능을 만들고 → 테스트하고 → 다음 기능으로
- **"이거 해줘" → "이것도 추가해줘" → "이건 안 되네, 고쳐줘"** 패턴

**4. MVP(Minimum Viable Product) 접근**
- 처음부터 완벽한 시스템을 설계하지 않음
- **동작하는 최소 기능**을 먼저 만들고
- **사용하면서 필요한 기능을 추가**

#### 💡 결론

| 이유 | 설명 |
|------|------|
| **문제를 발견하면서 해결책을 찾아감** | 관계형 DB의 한계 → 그래프 DB 도입, 여러 페이지 분산 → SPA 통합 |
| **실제 사용하면서 필요한 기능을 파악** | 단순 폼 빌더 → 업무량 분석 → 역량 평가 → 성과 증명 |
| **유연한 확장이 가능** | 초기 설계에 얽매이지 않고 필요에 따라 구조 변경 |
| **Vibe Coding과의 시너지** | AI와의 대화형 개발은 본질적으로 점진적/반복적 |
